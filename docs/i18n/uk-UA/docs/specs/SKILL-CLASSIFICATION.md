## 🎯 Purpose

Класифікатор надає кожному навику**послідовний машинозчитуваний профіль**, перш ніж він потрапить до каталогу. Виконує чотири роботи:

1. 📋**Parse**— передній матеріал YAML і тіло розцінки
2. 🏷️**Нормалізувати**— Мітки категорій до канонічної таксономії
3. 📊**Класифікувати**— зрілість, найкращі практики, якість і оцінка безпеки
4. 📁**Видавати**— артефакти метаданих, які споживаються сценаріями збірки, документами та CI---

### 📋 Best Practices Score (0-100)

Евристика оцінює:

| Сигнал | Що перевіряє |
|:-------|:--------------|
| 📛 Якість слимака | форматування поля `name` |
| 📝 Опис | Чіткість, обсяг, інформативність |
| 📐 Структура | Розділи та ієрархія документа |
| 💡 Приклади | Кодові огорожі та приклади блоків |
| 🔗 Посилання | Пов’язані локальні `посилання/`, `скрипти/` та допоміжні пакети підтримки |
| 🧰 Працездатність | Приклади виконуваних локальних сценаріїв і конкретних фрагментів робочого процесу |
| 🧩 Глибина пакету підтримки | Кілька сімейств підтримки, багаторазові файли, метадані агентів і робочі активи |
| 🩺 Усунення несправностей | Явні пари «Симптоми» та «Рішення» |
| 📚 Покриття | Розділи `Коли використовувати`, `Найкращі методи`, `Усунення несправностей` і `Додаткові ресурси` |
| 🌐 Портативність | Інструментально-агностичне формулювання |
| 📅 Свіжість | Уникнення жорстко закодованих дат |

**Поточний рівень**

| Рівень | Діапазон балів |
|:-----|:-----------|
| `відмінно` | 90-100 |
| `добре` | 70-89 |
| `чесно` | 50-69 |
| `потрібна робота` | 0-49 |

Бомбардир навмисно**досить семантичний, щоб створити розповсюдження**між зрілими навичками. Навичка з чистою структурою може отримати хороші оцінки, але щоб досягти верхньої смуги, вона також потребує сигналів глибини, таких як:

- кілька прикладів, а не один
- кілька випадків усунення несправностей
- відповідні навички
- багатші локальні пакети підтримки
- більше ніж одна допоміжна сім'я, окрім простої прози, в ідеалі включаючи `агенти/` або `активи/`, де вони додають реальне повторне використання
- окремий розділ `## Workflow` із підрахунком кроків
- принаймні одна невелика робоча таблиця або карта рішень, якщо це покращує ясність виконання
- більше операційної специфіки, ніж звичайний шаблон
- чіткіша глибина робочого процесу та засоби підтримки прийняття рішень
- глибина пакета підтримки, яка виходить за межі одного файлу `references/` та одного пов’язаного сценарію
- достатня кількість файлів підтримки для багаторазового використання, щоб відчувати себе як набір для робочого процесу, а не як доповнення для однієї нотатки
- достатня робоча щільність, щоб відокремити відшліфований контур від багаторазового робочого набору

Це означає, що структурно завершена навичка все ще може опинитися у високих 90-х замість «100», якщо її пакет підтримки вужчий, його ресурси для прийняття рішень менші або його робоча щільність нижча, ніж у найсильніших навичок у каталозі.---
