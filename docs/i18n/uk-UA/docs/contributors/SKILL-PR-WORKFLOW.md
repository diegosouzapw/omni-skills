Це канонічний потік репозиторію для запитів на отримання, які додають або суттєво оновлюють один або кілька рідних навичок.

Використовуйте його, коли:

- додавання нового навику в `skills/`
- поглиблення пакета з новими предметними навичками
- доставка більшого пакета підтримки
- перевірка гілки за допомогою приватного розширювача перед тим, як супроводжуючі об’єднають її## Target Outcome

Сильна рідна навичка PR має:

- рідна навичка під `skills/`
- достатньо вмісту для публічного валідатора, щоб класифікувати та індексувати його
- проходження громадської перевірки та випробувань
- автоматична обробка енхансера під час ПР
- подальший PR `skills_omni/` після публікації розширених похідних
- рідна мова, збережена мовою оригіналу, якщо це необхідно
- підібраний покращений вихід, переписаний англійською мовою
- односторонній потік від нативного до підготовленого, який не повертає `skills_omni/` назад у нативний енхансер.## Enhancer Outcome States

Загальнодоступний PR-підсилювач тепер повідомляє про чотири стани, видимі супроводжуючим:

- `завершено`
  Покращену похідну було згенеровано чітко, і вона придатна для супутньої публікації `skills_omni/`.
- `деградував`
  Підсилювач завершив роботу, але використав резервний шлях або видав попередження. Перед тим, як розглядати похідну як справну, очікується перевірка супроводжуючої сторони.
- `заблоковано`
  Запуск було зупинено через проблеми з інфраструктурою або валідацією, як-от помилка попередньої перевірки хостингу OmniRoute або помилка перевірки кандидата, що має завадити публікації.
- `не вдалося`
  У розширювачі сталася неочікувана помилка під час виконання, тому її слід перевірити.## Recommended Branch Shape

Створіть фокусну гілку:```bash
git checkout -b feat/<short-skill-theme>
```

приклади:

- `feat/incident-observability-evals`
- `feat/devops-skill-pack`
- `feat/security-skill-pack`## Native Intake Rules

Поверхня громадського водозабору навмисно дозволена.

мінімум:```text
skills/<skill>/
└── SKILL.md
```

Рекомендовано, але більше не потрібно для прийому:```text
skills/<skill>/
├── SKILL.md
├── agents/openai.yaml
├── references/checklist.md
├── references/<support-file>.md
├── examples/<worked-example>.md
└── scripts/render_<artifact>.py
```

Нативний внесок може бути приблизним, неповним або виходити за рамки звичайного стандарту пакета підтримки. Це навмисно. `skills/` — це рідна поверхня всмоктування, а не керована похідна поверхня.

Мовна політика:

- рідна мова під `skills/` може бути написана будь-якою мовою
- розширювач зберігає нативний знімок у тому вигляді, в якому він поданий для перевірки походження
- підібрана похідна в `skills_omni/` завжди має бути написана англійською мовою

Більш сувора редакційна заборона тепер поширюється на:

- згенеровані метадані та перевірки безпеки
- огляд приватного енхансера
- похідна подальша підготовка під `skills_omni/`## Authoring Sequence

1. Створіть `skills/<skill>/SKILL.md`.
2. Додайте фронтматер, якщо можете, але відсутній або неповний фронтматер більше не блокує нативне споживання самостійно.
3. Додайте `agents/`, `references/`, `examples/` і `scripts/`, якщо вони вже є.
4. Оновіть `data/bundles.json`, якщо навик поглиблює існуючий пакет.
5. Відкрити PR. Решту тепер зробить автоматизація репо.## Validation Sequence

Співавтори можуть виконати цю точну послідовність перед відкриттям PR:```bash
npm run validate
npm run build
npm test
git diff --check
```

Якщо зміна також впливає на час виконання або поведінку пакування, також запустіть:```bash
npm run smoke
```

## What Happens Automatically During the PR

Коли PR відкривається або синхронізується та торкається лише рідних файлів прийому навичок у `skills/` плюс необов’язковий `data/bundles.json`, публічне репо тепер автоматично запускає приватний покращувач.

Поточний автоматизований потік:

1. Загальнодоступний робочий процес «Перевірити навички» працює на PR і перевіряє перевірку, збірку, згенеровані артефакти та тести.
2. Загальнодоступний робочий процес `Enhance PR Skills` запускається паралельно та обробляє змінені нативні навички одну за одною в режимі `live`.
3. Enhancer зчитує нативний навик із `skills/`, досліджує поточні найкращі практики та записує перевіреного покращеного кандидата в приватну робочу область.
4. Підсилювач зберігає знімок вихідного потоку вихідною мовою, коли це необхідно, але переписує підібраний вихід англійською.
5. Дописи покращувача повертаються до вихідного PR.
6. Enhancer оновлює коментар статусу PR після кожної обробленої навички з підсумками партії та останнім станом.
7. Після завершення він матеріалізує розширену похідну в `skills_omni/` і відкриває або оновлює супутній PR у загальнодоступному репо лише для `завершених` і `понижених` результатів.
8. Після того, як PR буде об’єднано в `main`, опитувальник із підтримкою приватного репо повторно обробляє будь-які змінені нативні навички, оновлює `workspace/enhanced/skills/<skill>/` і зберігає приватну розширену базову лінію у відповідності з останнім публічним нативним джерелом.
9. Після злиття робочий процес публічного випуску змінює версію пакета npm, повторно генерує артефакти каталогу, публікує випуск і автоматично позначає нову версію.

Ліміт тарифу:

- PR-підсилювач зараз обробляє**1 навик за хвилину**
- PR із 40 новими навичками може залишатися в черзі покращувача приблизно 40 хвилин
- PR показує, що робота як незавершений CI-запуск плюс коментар до прогресу, який розвиває навичку за навичкою

Автору не потрібно запускати покращувач вручну.## No-Loop Rule For `skills_omni/`

Підготовлена поверхня навмисно одностороння:

- нативний вхід вводиться через `skills/`
- приватний підсилювач перевіряє цей рідний вхід
- підібраний результат пропонується в `skills_omni/`
- `skills_omni/` більше ніколи не розглядається як нативне споживання
- пізніші рідні оновлення все ще повторно вводяться через `skills/` і замінюють приватну розширену базову лінію після повторної обробки

Репозиторій тепер забезпечує дотримання цієї межі:

- прямі публічні PR, які змінюють `skills_omni/` відхиляються
- туди приймаються лише партнерські PR, створені автоматизацією, із сімейства гілок `skills-omni/pr-*`
- змішані PR, які намагаються змінити як `skills/`, так і `skills_omni/` одночасно, відхиляються## Automatic Versioning After Merge

Злиття навичок до `main` тепер автоматично запускають робочий процес випуску сховища.

Поточна політика версії пакета:

- патч збільшується на `+1` для кожного відповідного злиття
- `0.0.1` → `0.0.2` → ... → `0.0.10`
- після `.10` пакунок переходить до наступного мінора та скидає патч
- `0.0.10` → `0.1.0`
- `0.1.10` → `0.2.0`

Шляхи запуску поточного випуску:

- `навички/**`
- `skills_omni/**`
- `data/bundles.json`

Це завдання автоматичного випуску:

1. обчислює наступну версію пакета з `package.json`
2. зустрічає `package.json` і `package-lock.json`
3. регенерує `metadata.json`, `skills_index.json`, `dist/` і `docs/CATALOG.md`
4. запускає конвеєр перевірки суворого випуску
5. повертає версії до `main`
6. створює тег Git для нової версії
7. публікує артефакти npm і GitHub Release

Важлива примітка щодо випуску:

- GitHub реєструє новий файл робочого циклу як активний робочий процес сховища лише після того, як цей файл досягне гілки за замовчуванням.
- Поки `Enhance PR Skills` не приземлиться на `main`, учасники можуть прочитати задокументований процес, але GitHub ще не виконуватиме цей робочий процес автоматично на публічних PR-ах.
- Після того, як робочий процес об’єднано в `main`, поведінка, описана вище, стає шляхом за замовчуванням для майбутніх рідних навичок PR.## Native vs Enhanced

Це репо тепер має дві різні поверхні:

- `навички/`
  Рідний забір. Це зберігає оригінальний внесок у тому вигляді, в якому він був поданий.
- `skills_omni/`
  Покращений похідний результат Omni, запропонований автоматизацією та підтримується командою Omni Skills Team.

Правила атрибуції для `skills_omni/`:

- розширена похідна стає Omni-збереженою
- оригінальний учасник і навичка, що надійшла вгору, залишаються зарахованими
- кожен розширений каталог зберігає файл `ATTRIBUTION.md` із початковим шляхом, PR, автором і контекстом джерела
- кожен розширений каталог є лише кураторським виходом і не повинен повторно надсилатися в рідний шлях прийому покращувача
- очікується, що кожен розширений каталог буде виведений англійською мовою, навіть якщо вихідний вихідний вихідний код не був таким## Manual Maintainer Commands

Автоматизація охоплює звичайне споживання PR, але супроводжувачі все ще можуть запускати приватний покращувач вручну, коли це необхідно.

Пакет проти гілки diff:```bash
python3 /path/to/omni-skills-private/scripts/enhance_repo_changes.py \
  --repo-root /path/to/omni-skills \
  --base-ref main \
  --head-ref HEAD \
  --mode live \
  --min-skill-interval-seconds 60 \
  --no-update-state
```

Огляд однієї навички:```bash
python3 /path/to/omni-skills-private/scripts/run_enhancer.py \
  --skill <skill-id> \
  --mode live \
  --source-reason pr-review \
  --source-ref HEAD
```

Очікувані результати підсилювача на навик:

- `workspace/incoming/original/<run-id>/<skill>/`
- `workspace/enhanced-candidates/<run-id>/<skill>/`
- `workspace/reports/<run-id>/research.json`
- `workspace/reports/<run-id>/rewrite.json`
- `workspace/reports/<run-id>/validation.json`
- `workspace/reports/<run-id>/score-delta.json`
- `workspace/reports/<run-id>/review.md`
- `workspace/reports/<run-id>/research-prompt.md`
- `workspace/reports/<run-id>/rewrite-prompt.md`## PR Body Expectations

Орган PR має зазначити:

- які навички були додані або вдосконалені
- які групи або робочі процеси вони поглиблюють
- яка перевірка виконана
- чи запускався автоматизований енхансер
- чи відкрив або оновив супутній PR `skills_omni/`
- будь-які виняткові примітки супроводжуючого щодо атрибуції або подальшого очищення## Reviewer Checklist

- нативний прийом є законним і нешкідливим
- оновлено створені метадані та маніфести
- оновлення пакетів є навмисними
- результати публічної перевірки та збірки зелені
- коментар статусу покращувача відповідає фактично зміненим навичкам і кінцевому результату
- будь-який супутній PR `skills_omni/` зберігає атрибуцію належним чином## Example PR Scope

Сильний приклад PR може додати тематичний набір, наприклад:

- один навик спостереження або DevOps
- один інцидент або навик безпеки
- один навик ШІ оцінки або підказки

Це достатньо, щоб скористатися системою оцінювання результатів, автоматичним підсилювачем, потоком публікацій `skills_omni/`, пакетами та моделлю атрибуції, не перетворюючи PR на повне переписування каталогу.
