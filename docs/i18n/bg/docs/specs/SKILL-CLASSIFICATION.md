## 🎯 Purpose

Класификаторът дава на всяко умение**последователен машинночетим профил**, преди да достигне до каталога. Изпълнява четири задачи:

1. 📋**Parse**— YAML преден материал и маркдаун тяло
2. 🏷️**Нормализиране**— Етикети на категории към канонична таксономия
3. 📊**Класифициране**— Зрялост, най-добри практики, качество и точкуване за сигурност
4. 📁**Излъчване**— артефакти на метаданни, консумирани от скриптове за компилация, документи и CI---

### 📋 Best Practices Score (0-100)

Евристиката оценява:

| Сигнал | Какво проверява |
|:-------|:---------------|
| 📛 Качество на охлюв | форматиране на полето `name` |
| 📝 Описание | Яснота, дължина, информативност |
| 📐 Структура | Секции и йерархия на документа |
| 💡 Примери | Кодови огради и примерни блокове |
| 🔗 Референции | Свързани локални `референции/`, `скриптове/` и помощни средства за пакети за поддръжка |
| 🧰 Оперативност | Изпълняеми примери за локален скрипт и конкретни фрагменти от работен процес |
| 🧩 Дълбочина на пакета за поддръжка | Множество семейства за поддръжка, файлове за многократна употреба, метаданни на агенти и оперативни активи |
| 🩺 Отстраняване на неизправности | Изрични двойки „Симптоми“ и „Решение“ |
| 📚 Покритие | Секции `Кога да се използва`, `Най-добри практики`, `Отстраняване на неизправности` и `Допълнителни ресурси` |
| 🌐 Преносимост | Агностична формулировка |
| 📅 Свежест | Избягване на твърдо кодирани дати |

**Текущо ниво**

| Ниво | Диапазон на резултатите |
|:-----|:-----------|
| `отличен` | 90-100 |
| `добър` | 70-89 |
| `справедливо` | 50-69 |
| `нуждае се от работа` | 0-49 |

Резултатът е умишлено**достатъчно семантичен, за да създаде разпространение**в зрели умения. Умение с изчистена структура може да даде добри резултати, но за да достигне горната лента, то също се нуждае от сигнали за дълбочина като:

- множество примери, не само един
- множество случаи на отстраняване на неизправности
- свързано ръководство за умения
- по-богати местни пакети за поддръжка
- повече от едно поддържащо семейство извън обикновената проза, в идеалния случай включително „агенти/“ или „активи/“, където те добавят истинска повторна употреба
- специален раздел „## Работен процес“ с броими стъпки
- поне една малка оперативна таблица или карта на решенията, когато подобрява яснотата на изпълнение
- повече оперативна специфика от обикновен шаблон
- по-ясна дълбочина на работния процес и активи за подпомагане на вземането на решения
- дълбочина на пакета за поддръжка, който надхвърля един `references/` файл и един свързан скрипт
- достатъчно многократно използвани файлове за поддръжка, за да се чувствате като комплект за работен процес, а не като добавка за една бележка
- достатъчна оперативна плътност, за да се отдели полиран контур от комплект за многократна употреба

Това означава, че едно структурно завършено умение все още може да достигне високите 90 вместо „100“, ако неговият пакет за поддръжка е по-тесен, неговите активи за вземане на решения са по-тънки или неговата оперативна плътност е по-ниска от най-силните умения в каталога.---

### 🛡️ Security Score (0-100)

Защитният слой съчетава:

| Скенер | Винаги активиран | Какво прави |
|:--------|:--------|:-------------|
| 🔍**Статично**| ✅ Да | Сканира SKILL.md, пакетирани файлове и скриптове |
| 🦠**ClamAV**| ⚙️ По избор | Сканиране на зловреден софтуер чрез `clamscan` |
| 🔒**VirusTotal**| ⚙️ По избор | Хеш търсене (без качване) |

**Семейства правила за статичен скенер:**
- 🎭 Бързи модели на инжектиране и ексфилтрация
- 💣 Разрушителни команди на обвивката
- 🔑 Подозрителни пътища за идентификационни данни или OS
- ⚠️ Рискови скриптови примитиви (`shell=True`, `pickle.load`, `eval`, `extractall`)

## ⚙️ Workflow Integration

```bash

npm run taxonomy:report       # Show category drift suggestions
```

### 🪝 Optional Git Hooks

```bash
npm run hooks:install
```
