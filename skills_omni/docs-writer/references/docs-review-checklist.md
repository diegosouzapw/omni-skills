# Docs Review Checklist

Use this checklist before handing off documentation changes.

## Accuracy

- [ ] Every changed claim is supported by code, config, tests, CLI help, or trusted product specs.
- [ ] Commands, flags, paths, config keys, and defaults are current.
- [ ] Version-specific behavior is labeled clearly.
- [ ] Unverified details are marked as assumptions or open questions.

## Reader Success

- [ ] The intended audience is clear.
- [ ] The document type matches the user goal.
- [ ] Hidden prerequisites have been surfaced.
- [ ] The expected outcome is clear.
- [ ] Recovery notes are present where a step can fail.

## Writing Quality

- [ ] The page leads with the user goal.
- [ ] Language is plain, direct, and active.
- [ ] Terminology is consistent.
- [ ] Unnecessary repetition and filler have been removed.
- [ ] Sections are scannable and appropriately headed.

## Examples and Commands

- [ ] Examples are minimal but realistic.
- [ ] Placeholders are obvious and not misleading.
- [ ] Commands do not imply unsafe defaults without warning.
- [ ] Example output or expected results are described when helpful.

## Markdown and Structure

- [ ] Heading hierarchy is valid.
- [ ] Code fences use explicit languages where possible.
- [ ] Tables, lists, and callouts render cleanly.
- [ ] Relative links and anchors work.

## Change Impact

- [ ] Related docs were checked for drift.
- [ ] Navigation, sidebar, or index files were updated if needed.
- [ ] Changelog, migration, or upgrade docs were considered when relevant.

## Handoff

- [ ] Changed files are summarized.
- [ ] Verification performed is summarized.
- [ ] Remaining risks, assumptions, or follow-up questions are noted.
