# RFC Review Checklist

Use this checklist before approval or handoff.

## Scope and routing

- [ ] This is actually an RFC, not a TDD or ADR
- [ ] The decision is still open or the draft clearly records its review state
- [ ] The document is scaled appropriately for the impact level

## Metadata and lifecycle

- [ ] Status is explicit
- [ ] Owner / Driver is named
- [ ] Approvers or decision-makers are named
- [ ] Created and Last Updated dates are present
- [ ] Target Decision Date is present or intentionally marked TBD
- [ ] Supersedes / Superseded By is filled when relevant

## Context quality

- [ ] Current state is explained clearly
- [ ] Why now is explained
- [ ] Cost of inaction is described
- [ ] Affected stakeholders are identified

## Decision quality

- [ ] Decision criteria appear before options
- [ ] Criteria include weights or clear prioritization where useful
- [ ] There are at least two credible options
- [ ] Status quo is included when relevant
- [ ] Rejected options are treated fairly
- [ ] Recommendation is tied back to the criteria

## Evidence quality

- [ ] Important claims are supported by data, incidents, research, or concrete observations
- [ ] Assumptions are explicit
- [ ] Assumptions include confidence
- [ ] Invalidation triggers are recorded where useful

## Risk and execution readiness

- [ ] Major risks are described honestly
- [ ] Success metrics are measurable
- [ ] Follow-up actions are concrete
- [ ] Rollout / migration / rollback considerations are adequate for decision-making
- [ ] Implementation detail is not overwhelming the proposal

## Writing quality

- [ ] Title is specific and action-oriented
- [ ] Acronyms and overloaded terms are defined or obvious
- [ ] The draft avoids unsupported certainty
- [ ] The structure is easy to review

## Outcome management

- [ ] Open questions are visible
- [ ] Reviewers know what feedback is needed
- [ ] Status will be updated after the decision
