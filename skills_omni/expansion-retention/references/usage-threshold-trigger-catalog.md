# Usage Threshold Trigger Catalog

Use this file when designing expansion motions for usage, seats, modules, or overages.

## Principles

- expansion should follow realized value, not surprise pricing
- thresholds should be visible to the customer before they are breached
- high-value accounts should have a human path, not only automated prompts
- overage communication should be plain-language and early

## Common trigger types

| Trigger type | Example threshold | Recommended action | Safeguard |
| --- | --- | --- | --- |
| Usage volume | 50%, 75%, 90% of included volume | In-app alert, email, or CSM review | Explain current usage and projected impact |
| Seat growth | 80%+ seat utilization or new team invites | Team-plan recommendation | Show admin controls and pricing clearly |
| Feature gate hits | Same premium gate hit 3+ times | Contextual upgrade prompt | Show concrete value, not generic paywall pressure |
| Module adoption | Strong usage in adjacent workflow | Cross-sell discovery | Confirm relevance before campaign enrollment |
| Spend velocity | Spend rising unusually fast | Proactive outreach | Add review before large invoice shock |
| Renewal proximity | Expansion conversation 90-120 days before renewal | Bundle ROI + roadmap + renewal options | Avoid last-minute surprise commercial changes |

## Suggested threshold actions

### 50% of threshold

- awareness message
- usage dashboard reminder
- educational content on optimizing value

### 75% of threshold

- compare current plan vs next best option
- clarify expected bill or contract impact
- offer admin controls, alerts, or usage optimization guidance

### 90% of threshold

- urgent but clear notice
- offer human review for strategic accounts
- show options: optimize, cap, or upgrade

### 100% and beyond

- apply contract logic consistently
- communicate grace period or overage logic plainly
- avoid coercive dark patterns

## For usage-based models, always check

- can customers monitor usage easily?
- do invoices explain what changed?
- are overages explained before they happen?
- is there a cap, approval path, or alerting mechanism where appropriate?
- are CS and finance aligned on exception handling?
