# Customer Success Automation Compliance Note

This file provides operational guardrails, not legal advice.

## Use this note when

- recommending onboarding, re-engagement, renewal, or win-back automation
- using behavioral data to score health or trigger outreach
- reacting to stakeholder changes or champion loss

## Guardrails

### 1. Use approved data sources only

Prefer:
- first-party product usage
- CRM records
- support system data
- billing and contract data
- customer-provided stakeholder information

Avoid unapproved scraping or covert monitoring practices.

### 2. Minimize personal data use

Use the least personal data needed to support the workflow.

Prefer account-level or role-level signals where possible instead of detailed personal profiling.

### 3. Respect messaging and suppression requirements

Automated outreach plans should account for:
- unsubscribe handling
- suppression lists
- frequency limits
- sender reputation and deliverability
- regional messaging requirements

### 4. Avoid surveillance-style recommendations

Do not recommend:
- scraping personal social profiles without approval
- monitoring individuals in ways the organization has not reviewed
- enrichment practices that exceed approved policy

If stakeholder-change detection is needed, prefer approved account-owner updates, CRM changes, or customer-confirmed contacts.

### 5. Require review when legal interpretation matters

Escalate when the recommendation depends on:
- lawful basis for behavioral processing
- international privacy requirements
- consent or legitimate-interest interpretation
- regulated-industry customer restrictions

## Safer alternatives

Instead of "monitor champions on social platforms," prefer:
- maintain stakeholder maps in CRM
- ask customers to confirm key contacts during QBRs
- track account engagement decline at aggregate level
- create role-based coverage, not single-person dependency
