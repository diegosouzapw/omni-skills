This is a practical reporting aid, not a substitute for the full standard.

| :---------------- | :------------------------------- |
| Keyboard access | 2.1.1 Keyboard, 2.1.2 No Keyboard Trap |
| Meaningful focus order | 2.4.3 Focus Order |
| Focus visibility | 2.4.7 Focus Visible, 2.4.11 Focus Appearance (Minimum) |
| Landmarks, headings, structure | 1.3.1 Info and Relationships, 2.4.6 Headings and Labels |
| Meaningful sequence | 1.3.2 Meaningful Sequence |
| Accessible names and roles | 4.1.2 Name, Role, Value |
| Status messages and announcements | 4.1.3 Status Messages |
| Labels, instructions, errors | 3.3.1 Error Identification, 3.3.2 Labels or Instructions, 3.3.3 Error Suggestion |

## Reporting notes

- Map issues only when the relationship is reasonably clear.
- Do not force every product flaw into a WCAG citation if the user-impact note is clearer.
- If multiple criteria apply, list the most direct ones first.
- Keep the issue narrative task-focused even when WCAG mapping is included.
