# Code Connect and Dev Mode Reference

Use this reference in late Phase 3 or Phase 4.

## Positioning

Code Connect is optional.

Only introduce it after:
- component names are stable
- component properties are stable
- token naming is aligned with code
- core variable bindings are validated

## Dev Mode readiness checklist

Before handoff, verify:
- component names match implementation vocabulary
- variable names are understandable to developers
- code syntax metadata is present where expected
- pages and component grouping are navigable
- unresolved hardcoded values are minimized or documented

## Code Connect guidance

Use Code Connect when it improves code/design matching, not as a substitute for stable design-system structure.

Good timing:
- after one or more representative components are finalized
- when the implementation team already has stable coded components

Poor timing:
- while token names are still changing
- before component APIs are agreed
- during early experimentation

## Common pitfalls

- treating Code Connect as a requirement before the system is stable
- using code/design linkage to hide naming inconsistencies
- skipping basic Dev Mode review because Code Connect exists
